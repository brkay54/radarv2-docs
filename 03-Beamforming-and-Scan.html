Beamforming and Scan Schedule¶

Scope

How AERIS-10 forms and steers its beam, and the chirp → elevation → azimuth scan

hierarchy that ScanController reproduces — including the timing budget that

sets the demo's scan rate.

1. Array and electronic steering¶

The array is a 16-element uniform linear array (ULA) with half-wavelength spacing

$d = \lambda/2$. For a steering angle $\theta_0$, the required inter-element phase

shift is

$$\Delta\phi = -k d \sin\theta_0 = -\pi\sin\theta_0 \quad\Rightarrow\quad \Delta\phi_\text{deg} = 180\sin\theta_0,$$

so the steering command inverts as $\theta_0 = \arcsin(\Delta\phi_\text{deg}/180)$.

The array factor for uniform weighting is

$$|AF(\theta)| = \left|\frac{\sin(N\psi/2)}{\sin(\psi/2)}\right|,\qquad \psi = kd\sin\theta + \Delta\phi.$$

Key facts radarv2 carries as geometry (not as an RF model):

  • Half-power beamwidth ≈ 6.3° broadside ($\theta_{3\text{dB}} \approx 0.886\lambda/(Nd\cos\theta_0)$).
  • Steering range ±33°, kept inside the grating-lobe-free limit with ~30% margin.
  • Phase quantization: ADAR1000 7-bit phase shifters, step 2.8125°, which sets a

worst-case quantization sidelobe floor near −29 dB for $N = 16$.

AERIS-10 precomputes a phase_differences[31] table — the inter-element Δφ for each

of the 31 elevation beam positions, symmetric about index 15 (broadside, 0°). radarv2

treats the ±33° steering range as a chosen grating-safe cap (a parameter, not a

quantity derived from a maximum phase) and computes the per-element phase forward

from the beam angle: $\Delta\phi_\text{deg} = 180\sin\theta_0$. So the table spans

index 0 = $+98°$ (at $+33°$) to index 30 = $-98°$ (at $-33°$), ~2.2° per position.

Resolved steering inconsistency (was ±160°)

The upstream's phase_differences table lists index extremes of ±160°, but

$\arcsin(160/180)=62.7°$ — which contradicts both the documented ±33° steering range

and the array's own $\Delta\phi_\text{deg}=180\sin\theta_0$ relation (which gives

±98° at ±33°). radarv2 drops the ±160° figures as an upstream inconsistency and uses

the forward relation: ±33° cap → ±98° edge phase, well within the 7-bit ADAR1000

range. The steering range is the authoritative parameter; the phase follows from it.

2. Scan hierarchy¶

A full scan is a nested loop: chirps build a dwell, dwells sweep elevation, elevation

sweeps step azimuth, and azimuth steps complete a revolution.

Per-dwell chirp burst¶

Each beam dwell fires M = 32 chirps as two bursts separated by a guard interval:

  • 16 long chirps at PRI₁ = 167 µs (toggling new_chirp each time).
  • Guard interval of 175.4 µs (lets the matched filter flush).
  • 16 short chirps at PRI₂ = 175 µs.

In hardware this is driven by STM32 GPIO toggles that the FPGA edge-detects:

new_chirp (PD8) per chirp, new_elevation (PD9) per elevation position (resets

the Doppler accumulator), new_azimuth (PD10) per azimuth step. In radarv2 these

become fields of ScanCommand: chirp_index, el_position, az_position, and a

frame_start flag at each dwell boundary that resets the DopplerProcessor's CPI

accumulator. See ScanCommand.

Coherent-integration caveat (upstream inconsistency I-1)

The upstream fires this 16-long + 16-short mixed burst as the "M = 32" dwell,

but its Doppler stage runs a 32-point coherent FFT, which needs 32

same-waveform, uniform-PRI chirps — the mixed burst can't supply that. radarv2

resolves it by modelling a dwell as 32 coherent same-waveform chirps

(waveform_type selects the mode), so the Doppler FFT is physically valid. Full

rationale in I-1.

3. Timing budget¶

The dwell time dominates everything; the FPGA pipeline latency (~266 µs) is ~21×

smaller than one dwell, so processing keeps up comfortably.

radarv2's canonical dwell (DEC-4)

The table above is the upstream reference hardware budget for the 16-long + 16-short

mixed burst ($T_\text{pos} ≈ 5.65$ ms). Because that mixed burst can't feed a 32-point

coherent Doppler FFT (the I-1 inconsistency), **radarv2's canonical dwell is 32 coherent

same-waveform (long) chirps** at $T_{r,1}=167$ µs → $T_\text{dwell} = 32 \times 167\,\text{µs}

= 5.344$ ms. This is the dwell the DopplerProcessor integrates and the value behind

$\Delta v = 2.67$ m/s (§5).

The mixed-burst budget is kept only as documented upstream fidelity.

How radarv2 reproduces the timing

ScanController is a tick-driven node. Rather than emit chirps at the real

167 µs PRI (which would flood the demo), it advances the scan state machine one

step per executor tick at a demo-friendly rate, preserving the structure

(chirp/elevation/azimuth nesting, long+short bursts, dwell boundaries) while

scaling the wall-clock rate. The prf and pulse_width it stamps into each

ScanCommand are the real values, so downstream resolution math stays correct.

4. Beam geometry in radarv2¶

For each ScanCommand, ScanController derives:

  • Beam azimuth from the azimuth position (0–49) and the platform yaw: each

position is 7.2° apart, referenced to true North via the magnetic declination

correction (−0.61°) and the platform heading from PlatformNav.

  • Beam elevation from the elevation position (0–30) via the

phase_differences[31] → $\theta = \arcsin(\Delta\phi_\text{deg}/180)$ map.

These angles are what RxSynthesizer uses to decide which scenario targets fall

inside the beam (within the HPBW) and at what aspect, and what CfarDetector /

HostEgress use to place a detection back into the world. The geometry chain is

radar polar → body → ENU → geodetic, detailed in

05-Node-Architecture and 06-Message-Catalog.

Monopulse DF refines the azimuth (shipped)

The beam-pointing azimuth above is a coarse estimate (~one beamwidth). radarv2 now

also models amplitude-comparison monopulse: RxSynthesizer forms two squinted

beams and stamps the sum/difference ratio as mono_az onto each IqFrame (carried

through RangeProfile/RangeDopplerMap), and CfarDetector inverts it through a

loadable ratio→angle calibration table (df_table) to refine a detection's azimuth

to sub-beamwidth accuracy. Full mechanism in 07-Detection-and-Tracking.
