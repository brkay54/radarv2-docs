radarv2 — Future Development¶

Purpose

Running list of post-v1 enhancements for the examples/radarv2 AERIS-10 demo.

Items here are not in the first reviewable slice; they are captured so the

initial architecture leaves room for them. Each entry states the idea, why it

matters, and the design hook it needs in the base implementation.

See 00-Index for the full doc set; FD-1's base seam is described in

08-Scenarios-and-Targets.

FD-1 — Realistic vehicle / target models in scenarios¶

Idea. Let a scenario declare a typed vehicle (e.g. a fighter jet, an

airliner, a helicopter, a cruise missile, a quadrotor) instead of a hand-scripted

position polyline. Each vehicle type carries realistic flight-dynamics parameters

and a maneuver algorithm so that, when a scenario places a "jet", it actually

flies plausible jet maneuvers — accelerations, turn rates, and speeds bounded by

real aircraft limits — and the radar pipeline sees a physically credible target.

Why it matters.

  • Stresses the detection + (future) tracking chain with realistic kinematics:

high-speed inbound, beam-aspect notch (near-zero radial velocity → MTI/Doppler

blind speed), coordinated turns, and rapid maneuvers that break constant-velocity

tracker assumptions.

  • Makes the demo a credible radar scenario sandbox, not just a fixed playback —

scenario authors compose engagements ("two jets in a pincer, one airliner

transiting") from a vehicle library.

  • Ties the synthetic RCS/Doppler return to vehicle class (a jet vs. a drone differ

in RCS, speed envelope, and maneuverability), exercising the CFAR thresholds and

range/velocity-resolution limits documented in the parameter tables.

Proposed shape.

  • A vehicle library: each vehicle type = { kinematic limits (v_min/v_max,

a_max, max load factor g, max turn rate ω_max, climb/dive limits), nominal RCS

(and aspect-dependent RCS model), motion model }.

  • Motion / maneuver models (graduated realism):
  • Waypoint + bounded kinematics — follow waypoints but clamp velocity and

acceleration to the vehicle envelope (coordinated-turn at ω_max).

  • Maneuver scripts — named maneuvers (straight-and-level, weave, break turn,

split-S, terrain-follow, orbit/loiter, attack run) parameterized per vehicle.

  • Closed-loop guidance — pure-pursuit / proportional-navigation toward a goal

or waypoint, integrating a 3-DoF point-mass flight model with the vehicle's

thrust/drag/turn limits. This is the "runs like a real jet" tier.

  • Standard target-motion models worth supporting for tracker realism:

constant-velocity (CV), constant-acceleration (CA), coordinated-turn (CT), and

the Singer maneuvering-target model — these are the canonical models the

IMM/tracking literature uses and align with the upstream tracking research notes.

Design hook needed in the base implementation.

  • The radarv2 scenario format (analogous to examples/radar's

scenarios/*.scenario.yaml codegen table) must allow a target entry to

reference a vehicle type id + initial state + a maneuver/route spec, not only

a precomputed trajectory. Keep the scenario→target sampling API one step removed

from "fixed positions" so a kinematic integrator can be dropped in later.

  • The target-generating node (the radarv2 analog of radar_bpp/scenario sampler)

should sample target state from a TargetMotion interface at dwell time, so the

bounded-kinematics / guidance integrators are pluggable behind one seam.

  • Vehicle RCS/Doppler should be a function of the type + geometry, so the

synthetic return reflects vehicle class.

Status: captured 2026-06-20. Not in the first review slice. Revisit after the

base AERIS-10 receive pipeline + a basic scripted-trajectory scenario are running.

FD-2 — Physical map layers (DTED terrain + road/coastline) in the demo¶

Idea. Load real physical map binaries — DTED (Digital Terrain Elevation

Data) tiles, karayolları haritası (road/highway maps), coastlines, or other

geographic binaries — into the radarv2 demo so the radar operates over, and is

visualized against, the actual world: terrain under the tracks in

Foxglove/Lichtblick, and terrain-aware behaviour in the pipeline.

Why it matters.

  • Realism in the pipeline: a DTED surface gives terrain masking and a

line-of-sight horizon — a target behind a ridge or below the radar horizon is

occluded, and ground returns become terrain-shaped clutter rather than a flat

noise floor. This stresses CFAR (07-Detection-and-Tracking) against real

clutter and makes scenarios (sea-skimmers,

terrain-following jets) behave credibly.

  • Visualization: rendering DTED + roads/coastline beneath the

TargetReport in Lichtblick is exactly the

"see those physical maps inside this demo" the operator wants — geographic

context for the air picture.

  • Geodetic grounding: the polar→ENU→geodetic transform (05-Node-Architecture)

gets a real backdrop to place targets on.

Proposed shape.

  • A terrain_provider / map_provider node loads DTED tiles + vector maps at

on_configure and serves: (a) a terrain-height + occlusion query to

RxSynthesizer/CfarDetector (masking, clutter), and (b) a static map layer to

the visualization (a one-shot MapLayer message or a Foxglove scene asset).

  • DTED is a well-defined binary (SRTM-style elevation grid, fixed posting); road/

coastline can be a simple packed polyline binary. Both are read once, cold path.

Blocked on a core capability — ASSET-01

aros has no proper node asset/file-loading mechanism today (no OSAL

filesystem primitive; the only file I/O is the quarantined MCAP replay path; a

raw ifstream in on_configure breaks the no-exceptions-in-core and

portability conventions). FD-2 therefore depends on ASSET-01 on the aros

roadmap (.planning/ROADMAP.md Backlog; full spec + code pointers in

aros/BACKLOG.md): an OSAL read primitive + manifest-resolved asset

dependencies + asset bundling. Interim: radarv2 may ship a Linux-only ifstream

loader, clearly marked non-portable, until ASSET-01 lands.

Status: captured 2026-06-20. Pairs naturally with FD-1 (terrain-following /

low-altitude vehicle maneuvers need the terrain surface).

Shipped (FD-2, R5 + this milestone)

The MapProvider half landed in R5 (synthetic ridge tile + MapLayer over SHM +

CfarDetector terrain masking + RS1 terrain-shaped clutter). FD-2 then made the

map proper and at scale:

  • Proper DTED — radarv2_utils/asset/dted.hpp parses the real MIL-PRF-89020B

image (UHL/DSI/ACC headers + per-column records of big-endian signed-magnitude

posts) into the ENU height grid; MapProvider bilinear-resamples it to the 64×64

tile (falling back to the ASSET-01 interim raw-int16 loader). Replaces the

raw-blob stand-in.

  • At-scale terrain — radarv2_utils/radar/terrain_gen.hpp fbmHeight is a

deterministic multi-octave (fBm) value-noise heightfield; MapProvider's

terrain_mode: fractal fills the whole tile with varied rolling relief instead

of one Gaussian ridge (ridge stays the default → no regression).

  • Land-cover clutter — the RS1 clutter model gains optional sea/land typing

(a patch at/below sea_level_m uses the weaker sea_gamma_db), so a coastline

tile returns strong land clutter and a quiet sea.

Still open (true ASSET-01): an OSAL readFile/MappedFile primitive +

manifest-resolved asset bundling (tracked as INF1); road/coastline vector

features beyond the single demo road; live Lichtblick terrain rendering (FD-5).

FD-3 — Radar model library (real professional radars, selectable & definable)¶

Idea. Refactor radarv2 from "model the one AERIS-10 hobby radar" into a

model-driven simulation: a library of real, recognizable professional radars

(≥ 5–6, specs sourced from the open literature) plus AERIS-10, each captured as a

radar definition (a parameter profile). A demo author selects a radar by model

name, and can define new ones by writing a profile — no code changes.

Why it matters. Makes radarv2 a credible radar sandbox: the same node graph runs

as an S-band naval phased array, an X-band AESA fire-control, an L-band volume-search

set, etc., just by swapping the model. Different bands/beamwidths/PRFs exercise the

DSP and detection chain across realistic regimes.

Proposed shape.

  • A radar definition schema (extends the per-node params): band/center frequency →

wavelength, bandwidth → range resolution, beamwidth → angular resolution, PRF/CPI →

velocity resolution + ambiguity, instrumented range → SNR budget, scan type

(mechanical rpm / electronic) + revisit rate, antenna type (PESA/AESA), array size.

  • Candidate library (real radars, public specs — see the web-research digest in

examples/radarv2/.design/): e.g. AN/SPY-1 (S-band PESA), AN/TPY-2 (X-band AESA),

Thales SMART-L (L-band), Ground Master 400 (S-band AESA), Saab Giraffe (C-band),

AN/APG-81 (X-band airborne AESA) — plus AERIS-10 (X-band hobby).

  • ScanController/RxSynthesizer/CfarDetector already read waveform + geometry from

params/ScanCommand; FD-3 generalizes those into a named model profile so the whole

pipeline is model-parameterized.

Design hook (v1): keep every physical constant (fc, B, beamwidth, PRF, Rmax, scan

rate) in the parameter profile, never hard-coded — so a model is data, not code.

Shipped (FD-3)

The library is realized: twelve profiles under config/radars/*.profile.yaml

(spy1 / tpy2 / smartl / gm400 / giraffe / apg81 / fps117 / aeris10 + the ASELSAN fleet

models kalkan2 / alp300 / alp100 / eirs), each selectable by model name from a fleet.

The library showcase fleet runs five across four bands together; radarv2 list radars

enumerates them, and the RadarProfileLibrary ctest gates every profile's schema +

instantiability. Full write-up: 10-Radar-Model-Library.

FD-4 — Multiple radars running together + data fusion + central control¶

Idea. Run several radar instances simultaneously (different models and/or

sites), fuse their detections/tracks into one common operating picture for the end

user, and control all radars centrally (start/stop, mode, scan sector) from one place.

Why it matters. This is the real operational value: a layered, multi-sensor air

picture. Fusion resolves the same target seen by multiple radars into one track with

better accuracy and coverage; central control commands the whole network.

Proposed shape.

  • Multi-instance deployment: instantiate the radar node graph N times (one DKM set

per radar instance, distinct diagtree roots /AERIS10/<radar_id>/…), each with its

own model profile + site pose. Static manifest (fits ADR-004) — N is a deploy choice.

  • FusionCenter node: subscribes to every radar's TargetReport/detections,

transforms each into a common ENU/geodetic frame, associates across radars (GNN/track-

to-track), and publishes a fused CommonPicture (unified tracks + which radars

contributed). This is the multi-radar analog of the existing examples/radar

associator/tracker, lifted to the inter-radar level.

  • ControlCenter node (or a control topic): broadcasts commands to all radars

(start/stop, waveform/mode, scan sector, revisit) and aggregates health — the central

operator seat.

  • Track prediction (CV/CA/IMM, pairs with FD-1) so the fused

picture shows predicted positions, not just last-seen.

Design hook (v1): define TargetReport in a common world frame (ENU/geodetic,

already the case) so fusion needs no per-radar special-casing; give each radar a unique

id namespace.

Shipped (FD-4, Linux-verified)

Built with two deliberate departures from the sketch above, both intentional:

  • Distributed, not centralized — every radar runs its own FusionCenter

(Covariance Intersection over its own + every peer's world-track reports) and its own

ControlCenter (closed-loop cue: pick the priority fused threat, emit a RadarControl

set_sector so the staring beam follows the fused picture). No single fusion/control

process to lose.

  • Dynamic fleet size (config, not compiled) — the FusionCenter reads a peer_tracks

list (filled by the spine generator) and maps each transport source_id to a

contributor bit; any fleet up to 32 works with no code change.

The tracking that feeds fusion (PlotConditioner → TrackManager, RT0–6) is also

shipped. Full write-up: 11-Multi-Radar-Fusion.

FD-5 — 3D visualization in Lichtblick (objects, beams, signals, detections, predictions)¶

Idea. A rich 3D scene in Lichtblick (the vendored tools/lichtblick) showing

the whole engagement: platform/radar sites, the beams (steered cones/wedges) and

signals, live detections, predicted tracks, and **target models rendered as

3D objects — fighters, helicopters, etc. — with multiple radar models** visible

together.

Why it matters. "See the objects in 3D" is the payoff: an operator-grade view of

beams sweeping, returns lighting up, tracks predicting ahead, across a multi-radar

network — far more legible than 2-D panels.

Proposed shape.

  • A scene-builder (a Lichtblick User Script, like examples/radar's

tracks_scene.user.ts) converting the fused CommonPicture + per-radar state into a

foxglove.SceneUpdate: radar sites as models, beams as translucent cone/wedge

primitives oriented by the current az/el, detections as points, tracks as labelled

arrows with predicted-position trails, and target meshes keyed by vehicle class

(jet/helicopter/airliner/drone — pairs with FD-1).

  • Multiple radars distinguished by colour/namespace; the DTED/road map (FD-2)

as the ground layer.

  • Drive it over the aros-foxglove-bridge, the same path the existing demo uses.

Design hook (v1): publish everything the scene needs as typed aros messages in a

world frame (poses, beam az/el, detections, tracks, vehicle class) so the scene

builder is a pure transform with no hidden state.

Status — PENDING (FD-5)

FD-3 (model library) and FD-4 (multi-radar fusion + control) are shipped; the rich

3-D Lichtblick scene is the remaining FD item. The data it consumes (per-radar beam

state, DetectionReport, tracks, fused CommonPicture, MapLayer) is already published

as world-frame messages, so the scene builder stays a pure transform when it lands. The

design is written up in 12-3D-Scene.

FD-6 — Foxglove schema bridge¶

Idea. A schema bridge that exposes radarv2's typed aros messages to Foxglove /

Lichtblick with their own well-known schemas (so panels bind natively rather than

through a hand-written User Script), pairing with the FD-5 3-D scene.

Status: planned. The current path serves Foxglove-native MCAP bags (radarv2 record /

replay) and a live aros-foxglove-bridge; FD-6 adds the schema mapping layer.

Loadable binary assets (shipped, cross-cutting)

Three build-time binary tables now load at on_configure like the DTED tile: the DTED

terrain tile (MapProvider), the matched-filter reference bank (PulseCompressor, param

matched_filter_table, generator scripts/gen_matched_filter_table.py), and the

monopulse ratio→angle DF table (CfarDetector, param df_table, generator

scripts/gen_df_table.py). Each falls back to in-code synthesis when absent. See

R9.
