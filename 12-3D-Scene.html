3D Scene Visualization (Lichtblick)¶

Scope

The design for a rich 3-D scene in Lichtblick showing the whole engagement —

radar sites, steered beams, live detections, predicted tracks, and

target meshes (fighters, helicopters, …) for multiple radars at once, over a

terrain ground layer. This is the FD-5

design. It consumes the CommonPicture and renders it

through the same Foxglove bridge the base demo already uses.

Status — PENDING (the last open FD item)

Everything this scene consumes is **already shipped and published as world-frame

messages**: the base pipeline, tracking, the model library (FD-3),

multi-radar fusion + control (FD-4), and the

DTED MapLayer (FD-2).

The 3-D Lichtblick scene builder itself is the remaining work; a native Foxglove schema

bridge is tracked as FD-6.

1. Why a 3-D scene¶

Two flat panels (a PPI sweep and a range-Doppler heatmap) tell you that a radar

detected something. They do not show where in the world, which beam saw it,

where it is going, or how three radars' coverage overlaps. A 3-D scene does:

beams sweep as translucent cones, returns light up along them, tracks extend arrows

toward their predicted positions, target meshes show vehicle class, and the whole

multi-radar network is legible at a glance over real terrain. "See the objects in 3-D"

is the payoff of the FD-3/4/5 arc.

2. The rendering path¶

The scene is built by a Lichtblick User Script — the same mechanism

examples/radar uses for tracks_scene.user.ts — that converts typed aros messages

into one foxglove.SceneUpdate. Nothing in the scene builder holds hidden state: it is

a pure transform from the world-frame messages to scene primitives, driven over the

aros-foxglove-bridge.

3. What gets drawn¶

Each scene element maps a message field to a Foxglove SceneEntity primitive:

Scene elementFoxglove primitiveDriven by
Radar sitemodel / cube + labelper-radar site pose
Steered beamtranslucent cone / wedgebeam az/el + HPBW from ScanCommand
Detectionsphere / point, sized by SNRDetectionReport cells
Trackarrow + label, coloured by qualityCommonPicture fused track
Predicted positiondashed line / trail ahead of tracktrack predicted state
Target bodymesh keyed by vehicle classtrack class (jet/heli/airliner/drone)
Groundtextured height mesh + road linesMapLayer (DTED + roads)
Radar coveragetranslucent coverage volumeper-radar range + sector
Beams are the most informative primitive

A steered beam drawn as a translucent cone oriented by the live `(azimuth,

elevation)` and opened to the half-power beamwidth makes the scan itself visible —

you watch the volume being searched, and a detection appears inside the cone that

found it. Multiple radars' cones in different colours show how the network's coverage

interleaves.

Multiple radars, one scene¶

Each radar's primitives live under its own SceneEntity namespace (its diagtree id),

coloured per radar, so three different models render together without collision —

distinct beam colours, distinct site markers, and a shared fused-track layer on top.

The terrain (FD-2

/ MapProvider) is the common ground layer beneath all of them.

4. Build order and dependencies¶

The scene is the last piece of the FD arc because it consumes everything else:

  • Base single-radar pipeline (v1) gives DetectionReport + beam state → a minimal

scene (one site, its beam, its detections) is already buildable on v1. (shipped)

  • FD-3 makes radars selectable → multiple distinct models. (shipped)
  • FD-4 gives CommonPicture → the fused-track layer. (shipped)
  • FD-1 gives

vehicle class + kinematics → meshes and credible predicted trails. (partial: dynamic motion shipped; vehicle meshes pending)

  • FD-2

gives MapLayer → the terrain ground layer. (shipped; live Lichtblick render pending)

A minimal scene ships with v1

Even before fusion and vehicles, the base pipeline can drive a one-radar scene (site +

live beam + detections over a flat ground), reusing examples/radar's

tracks_scene.user.ts pattern. The rich multi-radar scene is the FD-5 endpoint; the

seam — publish everything the scene needs as typed world-frame messages — is

established from v1 so the builder stays a pure transform.
