FPGA Signal Processing Pipeline¶

Purpose: Trace the complete receive-path signal flow from the AD9484 ADC input at 400 MHz through every FPGA processing stage to USB output, documenting clock domains, I/O formats, transformations, and magic number derivations for each stage.

Prerequisites: Familiarity with:

  • Symbol Table -- all symbols used herein
  • Parameter Table -- numerical values for both variants
  • Conventions -- equation formatting rules
  • FPGA Board -- clock domain architecture and resource capacity
  • Timing Budget -- end-to-end pipeline latency analysis
  • FMCW Theory -- beat frequency and range equations

1. Pipeline Overview¶

The AERIS-10 receive pipeline in radar_receiver_final.v processes digitized IF samples through ten stages, converting raw ADC data into range-Doppler maps with threshold detection. The pipeline is organized by signal flow -- each stage receives the output of the previous stage and produces input for the next.

3. Stage 1: ADC Acquisition¶

Modules: lvds_to_cmos_400m, ad9484_lvds_to_cmos_400m

Clock domain: ADC 400 MHz (LVDS)

The AD9484 ADC digitizes the IF signal at $f_s$ (see Parameter Table) with 8-bit resolution. The ADC outputs data on differential LVDS pairs along with a data clock output (DCO) at $f_s$.

3.1 LVDS-to-CMOS Conversion¶

The lvds_to_cmos_400m module recovers a single-ended 400 MHz CMOS clock (clk_400m) from the LVDS DCO differential pair (adc_dco_p, adc_dco_n). This recovered clock becomes the processing clock for the entire 400 MHz domain.

The ad9484_lvds_to_cmos_400m module converts the 8-bit LVDS data bus (adc_d_p[7:0], adc_d_n[7:0]) to single-ended CMOS (adc_data_cmos[7:0]) and generates data valid signaling (adc_valid). It also provides an ADC power-down control output (adc_pwdn).

3.2 Clock Domain Crossing to Processing Domain¶

The cdc_adc_to_processing module synchronizes 8-bit ADC data from the ADC DCO clock domain to the recovered 400 MHz processing clock domain using a 3-stage synchronizer (STAGES=3, WIDTH=8):

ParameterValue
Input clockadc_dco_cmos (ADC DCO)
Output clockclk_400m (recovered CMOS)
Data width8 bits
Synchronizer stages3
Outputcdc_data_cmos[7:0], cdc_valid

4. Stage 2: Digital Down-Conversion¶

Module: ddc_400m_enhanced

Clock domain: 400 MHz (NCO, mixer, CIC) transitioning to 100 MHz (FIR output)

The DDC translates the digitized IF signal from $f_\text{IF}$ (see Parameter Table) to complex baseband (I/Q) using an NCO-based quadrature mixer, followed by CIC decimation and FIR compensation filtering.

4.1 NCO Phase Increment Derivation¶

The NCO generates complex sinusoids at $f_\text{IF}$ using a 32-bit phase accumulator. The phase increment per clock cycle $\Delta\phi_\text{NCO}$ (defined in the Symbol Table) is:

$$

\Delta\phi_\text{NCO} = \frac{f_\text{IF}}{f_s} \times 2^{32} \tag{SW-1}

$$

Numerical verification (this derives the magic number PHASE_INC_120MHZ in ddc_400m.v):

$$

\Delta\phi_\text{NCO} = \frac{120 \times 10^6}{400 \times 10^6} \times 2^{32} = 0.3 \times 4{,}294{,}967{,}296 = 1{,}288{,}490{,}189 = \text{0x4CCCCCCD} \tag{SW-2}

$$

The NCO module nco_400m_enhanced uses this phase increment (frequency_tuning_word) to produce 16-bit signed sine and cosine outputs (sin_out[15:0], cos_out[15:0]).

4.2 LFSR Phase Dithering¶

To reduce spurious tones caused by phase truncation, an 8-bit linear feedback shift register (LFSR) module lfsr_dither_enhanced adds pseudo-random dither to the phase increment:

phase_inc_dithered = PHASE_INC_120MHZ + {24'b0, phase_dither_bits}

The LFSR uses the polynomial $x^8 + x^6 + x^5 + x^4 + 1$ (feedback taps at bits 7, 5, 4, 3) with a non-zero initial state. The 8-bit dither is zero-extended to 32 bits, perturbing only the least significant bits of the phase increment to spread phase truncation spurs across the noise floor.

4.3 Quadrature Mixer¶

The mixer multiplies the sign-extended ADC data by the NCO cosine (I channel) and sine (Q channel) outputs:

SignalWidthDescription
adc_signed18-bitADC data converted from unsigned 8-bit to signed, zero-extended
cos_out / sin_out16-bit signedNCO quadrature outputs
mixed_i / mixed_q34-bit signedMixer product ($18 \times 16 = 34$ bits)

The mixer includes saturation detection with a latched overflow counter (saturation_count) and real-time monitoring (mixer_overflow_i, mixer_overflow_q). A test mode (force_saturation) can inject full-scale values for verification.

4.4 CIC Decimation¶

Module: cic_decimator_4x_enhanced (two instances: I and Q channels)

Clock domain: 400 MHz

The CIC filter reduces the sample rate by $D_\text{CIC}$ (see Parameter Table) with $N_\text{CIC}$ cascaded integrator-comb stages. The DC gain $G_\text{CIC}$ (defined in the Symbol Table) is:

$$

G_\text{CIC} = D_\text{CIC}^{N_\text{CIC}} = 4^5 = 1024 = 2^{10} \tag{SW-3}

$$

This gain of $2^{10}$ is compensated by a right shift of 10 bits (>>> 10) in the output path, which is the derivation of the magic number "shift 10" in the CIC implementation.

The CIC includes saturation detection with latched overflow monitoring for diagnostics.

4.5 CDC Crossing: CIC to FIR¶

After CIC decimation, the 18-bit I/Q data at 100 MHz equivalent rate (still clocked by the 400 MHz domain) crosses to the 100 MHz system clock domain via two 3-stage synchronizers (CDC_FIR_i, CDC_FIR_q), each with WIDTH=18 and STAGES=3.

4.6 FIR Compensation Filter¶

Module: fir_lowpass_parallel_enhanced (two instances: I and Q channels)

Clock domain: 100 MHz

The FIR filter compensates for CIC passband droop (the $\text{sinc}^{N_\text{CIC}}$ magnitude response of the CIC filter). Dual I/Q channels are processed independently by separate filter instances (fir_i_inst, fir_q_inst).

ParameterValue
Input18-bit signed (from CDC)
Output18-bit signed (fir_i_out, fir_q_out)
Clock100 MHz system clock
Ready signalfir_i_ready, fir_q_ready
Overflow detectionfilter_overflow output

The output stage latches the FIR results on the negative edge of the 100 MHz clock to produce the final baseband I/Q outputs (baseband_i[17:0], baseband_q[17:0]).

5. Stage 3: DDC Output Interface¶

Module: ddc_input_interface

Clock domain: 100 MHz

The DDC output interface scales and aligns the 18-bit I/Q outputs from the DDC to 16-bit signed format for downstream processing:

6. Stage 4: Reference Chirp and Latency Buffer¶

Modules: chirp_memory_loader_param, latency_buffer_2159

Clock domain: 100 MHz

Before matched filtering, the reference chirp waveform must be loaded from memory and time-aligned with the signal data arriving from the FFT pipeline.

6.1 Chirp Memory Loader¶

The chirp_memory_loader_param module provides the stored reference chirp coefficients (16-bit I/Q) from block RAM. It supports both long and short chirp waveforms and accepts a segment select input for multi-segment overlap-save processing:

PortWidthDescription
segment_select2-bitSelects which of the 4 segments to load
mem_request1-bitTriggers a memory read
use_long_chirp1-bitSelects between long (3000 sample) and short (50 sample) reference
sample_addr10-bitAddress within the current segment (0--1023)
ref_i / ref_q16-bitReference chirp I/Q output
mem_ready1-bitIndicates data is available

6.2 Latency Buffer¶

Derivation of the 3187-cycle latency buffer:

The latency_buffer_2159 module (named for its original design but parameterized with LATENCY=3187) delays the reference chirp data to align it with the signal data emerging from the forward FFT. The delay compensates for the combined latency of the FFT pipeline stages in the matched filter processing chain:

ParameterValueVerilog Param
Data width32-bit (16-bit I + 16-bit Q packed)DATA_WIDTH=32
Delay3187 clock cyclesLATENCY=3187
Input{ref_i, ref_q} from chirp memory--
Output{delayed_ref_i, delayed_ref_q}--

The 3187-cycle delay at 100 MHz corresponds to $31.87~\mu\text{s}$ and accounts for the forward FFT, spectral multiplication, and inverse FFT pipeline latency in the matched_filter_processing_chain sub-module. For end-to-end pipeline latency analysis, see 02_hardware/07_timing_budget.md.

7. Stage 5: Matched Filter (Pulse Compression)¶

Module: matched_filter_multi_segment

Clock domain: 100 MHz

The matched filter performs frequency-domain pulse compression using the overlap-save method. The signal is correlated with the stored reference chirp to produce a range profile with pulse compression gain proportional to the time-bandwidth product (see Eq. (LFM-7) in 01_physics/02_lfm_waveform.md for pulse compression theory).

7.1 Overlap-Save Segmentation¶

For long chirps, the $N_\text{FFT}$-point FFT buffer (see Parameter Table) cannot hold all samples in a single transform. The overlap-save method divides the input into $N_\text{seg}$ segments (defined in the Symbol Table):

ParameterSymbolLong ChirpShort ChirpVerilog Param
Total input samples--300050LONG_CHIRP_SAMPLES, SHORT_CHIRP_SAMPLES
FFT/buffer size$N_\text{FFT}$10241024BUFFER_SIZE
Overlap length$L_\text{overlap}$128--OVERLAP_SAMPLES
Segment advance$L_\text{adv}$896--SEGMENT_ADVANCE = BUFFER_SIZE - OVERLAP_SAMPLES
Number of segments$N_\text{seg}$41LONG_SEGMENTS, SHORT_SEGMENTS

The segment count for long chirps is derived from:

$$

N_\text{seg} = \left\lceil \frac{3000 - L_\text{overlap}}{L_\text{adv}} \right\rceil = \left\lceil \frac{3000 - 128}{896} \right\rceil = \left\lceil 3.205 \right\rceil = 4 \tag{SW-4}

$$

For short chirps, the 50 input samples are zero-padded to fill the full 1024-point buffer and processed as a single segment.

7.2 State Machine¶

The matched filter operates through an 8-state FSM:

StateNameDescription
0ST_IDLEAwaits chirp_start_pulse from STM32 mc_new_chirp toggle; sets total_segments based on use_long_chirp
1ST_COLLECT_DATACollects DDC samples into the input buffer; for long chirps, transitions when $L_\text{adv}$ new samples collected; for short chirps, transitions after 50 samples
2ST_ZERO_PADShort chirp only: fills remaining buffer positions (50--1023) with zeros
3ST_WAIT_REFRequests reference chirp segment via segment_request and mem_request; waits for mem_ready
4ST_PROCESSINGFeeds buffered samples and reference data to matched_filter_processing_chain (forward FFT, spectral multiply, inverse FFT)
5ST_WAIT_FFTWaits for matched_filter_processing_chain to complete (latency determined by FFT pipeline)
6ST_OUTPUTEmits range profile I/Q (pc_i, pc_q, pc_valid); checks if more segments needed
7ST_NEXT_SEGMENTOverlap-save: copies last $L_\text{overlap}$ samples to buffer start, resets write pointer to $L_\text{overlap}$; advances current_segment counter

7.3 Processing Chain¶

The matched_filter_processing_chain sub-module performs the core frequency-domain operations:

  • Forward FFT -- $N_\text{FFT}$-point FFT of the buffered input segment
  • Spectral Multiplication -- element-wise complex multiply with the FFT of the reference chirp
  • Inverse FFT -- $N_\text{FFT}$-point inverse FFT to produce the time-domain range profile

The processing chain receives 16-bit I/Q input (adc_data_i, adc_data_q) and the reference chirp (long_chirp_real/imag, short_chirp_real/imag), and outputs the range profile (range_profile_i[15:0], range_profile_q[15:0]) with a valid flag. The matched filtering implements the cross-correlation in the frequency domain (see Eq. (FMCW-1) in 01_physics/01_fmcw_theory.md for the beat frequency interpretation of range).

7.4 Input Data Formatting¶

The DDC output (16-bit from ddc_input_interface) is sign-extended to 18 bits before entering the matched filter:

.ddc_i({{2{adc_i_scaled[15]}}, adc_i_scaled}),  // 16-bit -> 18-bit sign extension
.ddc_q({{2{adc_q_scaled[15]}}, adc_q_scaled}),

Within the buffer, samples are truncated with rounding from 18-bit to 16-bit: ddc_i[17:2] + ddc_i[1].

$$

D_\text{rb} = \frac{N_\text{FFT}}{N_\text{rb}} = \frac{1024}{64} = 16 \tag{SW-5}

$$

9. Stage 7: Doppler Processing¶

Module: doppler_processor_optimized

Clock domain: 100 MHz

The Doppler processor computes a $N_\text{Doppler}$-point FFT across $M$ chirps for each of $N_\text{rb}$ range bins to measure target velocity. This implements the slow-time Doppler processing described in Eq. (FMCW-4) of 01_physics/01_fmcw_theory.md.

9.1 Memory Organization¶

The Doppler processor accumulates range bin data from $M$ chirps in block RAM before processing:

ParameterSymbolValueVerilog Param
Doppler FFT size$N_\text{Doppler}$32DOPPLER_FFT_SIZE
Range bins$N_\text{rb}$64RANGE_BINS
Chirps per frame$M$32CHIRPS_PER_FRAME
Memory depth$N_\text{rb} \times M$2048 entriesMEM_DEPTH
Data width--16-bit per channelDATA_WIDTH

Memory uses chirp-major addressing:

$$

\text{addr} = \text{chirp\_index} \times N_\text{rb} + \text{range\_bin} \tag{SW-6}

$$

The ( ram_style = "block" ) attribute directs Vivado to implement the I/Q memory arrays in FPGA Block RAM (see $N_\text{BRAM}$ in the Symbol Table).

9.2 Hamming Window¶

Before the Doppler FFT, each slow-time sample is multiplied by a Hamming window $w[n]$ (defined in the Symbol Table) in Q15 fixed-point format. The 32 pre-calculated coefficients are stored as 16-bit unsigned values:

IndexValue (hex)IndexValue (hex)
00x0800160x7FFF (peak)
10x0862170x781D
20x09CB180x69C4
...(symmetric)......
150x781D310x0862

The windowed samples are computed with Q15 multiplication and rounding:

mult_i = $signed(doppler_i_mem[addr]) * $signed(window_coeff[index]);
fft_input_i = (mult_i + (1 << 14)) >>> 15;  // Round to nearest

9.3 Doppler FFT¶

The Doppler FFT uses the Xilinx xfft_32 IP core (32-point forward FFT, AXI-Stream interface):

9.6 Frame Synchronization¶

In radar_receiver_final.v, frame synchronization detects transitions in chirp_counter to generate new_chirp_frame pulses. The current implementation uses Method B: frame start at chirp_counter == 0 or chirp_counter == 32, supporting 32-chirp frames within a 64-chirp sequence.

10. Stage 8: Threshold Detection¶

Module: Inline logic in radar_system_top.v

Clock domain: 100 MHz

CRITICAL: This is a placeholder threshold detector, NOT a true CFAR implementation. The variable names in the Verilog source (rx_cfar_detection, cfar_valid, cfar_counter) use CFAR terminology, but the actual implementation is a fixed magnitude threshold. A true CFAR algorithm (CA-CFAR, OS-CFAR, etc.) is documented as a future enhancement -- see Phase 5 (SWRES-01) and detection theory in 01_physics/02_lfm_waveform.md.

10.1 Implementation¶

The detection logic operates on the Doppler output from radar_receiver_final:

Doppler output format: {rx_doppler_imag[15:0], rx_doppler_real[15:0]}

The detector computes an approximate magnitude using the L1 norm (Manhattan distance) and compares against a fixed threshold:

$$

I| +| Q| > 10{,}000 \tag{SW-7}

$$

where $I$ and $Q$ are the 16-bit signed Doppler output components (rx_doppler_real, rx_doppler_imag). The absolute values are computed by conditional negation:

wire [16:0] mag = (rx_doppler_real[15] ? -rx_doppler_real : rx_doppler_real) +
                  (rx_doppler_imag[15] ? -rx_doppler_imag : rx_doppler_imag);

if (mag > 17'd10000) begin
    rx_cfar_detection <= 1'b1;  // Detection flag (misleading name)
    rx_cfar_valid <= 1'b1;
end

11. Stage 9: USB Output¶

Module: usb_data_interface

Clock domain: FT601 100 MHz (asynchronous to system clock)

The USB interface transmits range profiles, Doppler spectra, and detection flags to the host PC via the FT601 USB 3.0 controller in Slave FIFO mode. The FT601 clock ft601_clk_in at 100 MHz is generated by the FT601's own oscillator and is asynchronous to the system 100 MHz clock.

For full USB protocol documentation including packet format, state machine, and host-side interface, see 04_usb_protocol.md (SWDOC-04).

12. STM32-to-FPGA Control Interface¶

The STM32 microcontroller controls the FPGA pipeline timing through GPIO toggle signals. Each signal uses edge detection (rising edge of toggle) in the FPGA to generate single-cycle pulses.

12.1 Control Signals¶

SignalGPIO PinDirectionFunction
stm32_new_chirpPD8STM32 -> FPGAToggle on each new chirp transmission; triggers matched filter FSM (chirp_start_pulse)
stm32_new_elevationPD9STM32 -> FPGAToggle on elevation beam position change; resets Doppler accumulation for new beam
stm32_new_azimuthPD10STM32 -> FPGAToggle on azimuth position change; signals new scan column
stm32_mixers_enablePD11STM32 -> FPGAEnables/disables RF mixer path in DDC; controls NCO phase_valid
FPGA resetPD12STM32 -> FPGAActive-low reset; firmware toggles during initialization sequence

12.2 Pipeline State Transitions¶

The control signals drive the pipeline through the radar scan sequence:

  • Chirp start (stm32_new_chirp toggle): The matched filter FSM transitions from ST_IDLE to ST_COLLECT_DATA, beginning sample collection for the current chirp
  • Chirp completion: After $M$ chirps per beam position, the Doppler processor has accumulated a full frame and begins FFT processing across all $N_\text{rb}$ range bins
  • Elevation change (stm32_new_elevation toggle): Signals that the beam has moved to a new elevation position; the pipeline prepares for a new set of $M$ chirps
  • Azimuth change (stm32_new_azimuth toggle): Signals stepper motor has moved to the next azimuth position; begins a new elevation scan at the new azimuth

For the complete chirp sequence timing and scan schedule, see 02_hardware/07_timing_budget.md.
