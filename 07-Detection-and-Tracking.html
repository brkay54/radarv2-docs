Detection and Tracking¶

Scope

The detection theory behind CFAR, the CA-CFAR design radarv2 implements (upgrading

AERIS-10's placeholder threshold), and the tracking roadmap (deferred to future

work, with the upstream's own research as the reference).

1. Detection theory¶

Detection is a binary hypothesis test per range-Doppler cell: $H_0$ (noise only) vs.

$H_1$ (target + noise). The Neyman-Pearson optimal test compares the cell statistic

$T = |x|^2$ against a threshold $\gamma$ chosen for a target false-alarm probability:

$$P_\text{fa} = e^{-\gamma/\sigma_n^2} \quad\Rightarrow\quad \gamma = -\sigma_n^2 \ln P_\text{fa}.$$

The problem: $\sigma_n^2$ (noise power) is unknown and varies across the map (clutter,

gain, range). A fixed threshold therefore yields an uncontrolled false-alarm

rate — exactly the failure mode of AERIS-10's shipped detector.

The upstream detector is a placeholder

The FPGA's stage-9 detector is |I| + |Q| > 10000 — a fixed L1-magnitude

threshold with cfar_* signal names but **no reference cells and no adaptive

threshold**. The upstream documents a real CFAR as the intended replacement.

radarv2 implements that replacement directly.

Target fluctuation — Swerling models¶

The required SNR for a given $P_d$ depends on how the target's RCS fluctuates. The

Swerling models (RCS distribution × decorrelation rate) are the standard taxonomy, and

set how much margin a detector needs over the non-fluctuating ideal:

CaseRCS PDFDecorrelationExample

A Rayleigh (Swerling I) target costs ≈ +8 dB over the non-fluctuating case at

$P_d=0.9,\,P_\text{fa}=10^{-6}$; the per-case required-SNR figures are in

appendix R4b. radarv2's scenario

targets carry an RCS and (future, FD-1)

a fluctuation class, so the synthesizer can exercise these margins.

2. CA-CFAR (radarv2's detector)¶

Cell-Averaging CFAR estimates the local noise power from $N_\text{ref}$ reference

cells surrounding the cell under test (CUT), separated by $N_\text{guard}$ guard cells

that keep target energy from leaking into the estimate:

$$\hat P_n = \frac{1}{N_\text{ref}}\sum_{i=1}^{N_\text{ref}} T_i,\qquad \text{declare detection if } T_\text{CUT} > \alpha\,\hat P_n.$$

The scaling constant $\alpha$ is set from the desired false-alarm probability and the

window size:

$$P_\text{fa} = \left(1 + \frac{\alpha}{N_\text{ref}}\right)^{-N_\text{ref}}\quad\Rightarrow\quad \alpha = N_\text{ref}\left(P_\text{fa}^{-1/N_\text{ref}} - 1\right).$$

CA-CFAR loss (the SNR penalty vs. a known-noise threshold) shrinks as the window

grows: ≈ 1.5 dB at $N_\text{ref} = 16$, ≈ 0.8 dB at 32, ≈ 0.4 dB at 64 (at $P_d = 0.9$,

$P_\text{fa} = 10^{-6}$). radarv2 defaults to $N_\text{ref} = 32$, $N_\text{guard} = 4$,

$P_\text{fa} = 10^{-6}$ — the upstream's recommended starting point.

$N_\text{ref}$ counts total cells, not per-side

$N_\text{ref} = 32$ is the total number of reference cells averaged (16 on each

side of the CUT); $N_\text{guard} = 4$ is likewise total (2 per side). The

$\alpha = N_\text{ref}(P_\text{fa}^{-1/N_\text{ref}} - 1)$ and CFAR-loss formulas both

use the total. Reading 32 as per-side would double the averaging window and halve

the threshold scale — an easy off-by-2× $P_\text{fa}$ bug; the implementation asserts

the total convention.

Cell → physical conversion¶

Each detected cell (range_bin, doppler_bin) is converted using the values stamped

into the RangeDopplerMap:

  • range = range_bin × range_res;
  • radial velocity = (doppler_bin − doppler_bins/2) × velocity_res (centered, so

the zero-Doppler bin is stationary) — **valid because the DopplerProcessor

fftshifts the map** so bin 16 = DC (04-Signal-Processing-Pipeline); on a raw

(un-shifted) FFT, DC would sit at bin 0 and this formula would be wrong;

  • azimuth / elevation = the beam geometry from the dwell — **refined in azimuth

by monopulse direction finding** when a calibration table is loaded (see §2.1);

  • SNR = cell magnitude relative to the CFAR noise estimate.

Variants (selectable — RS2)¶

All four variants now ship as a per-radar parameter (cfar_variant: ca|go|so|os

on CfarDetectorParams); CA is the default, so the prior behaviour is unchanged.

The OS rank is os_rank (1-based, -1 → 0.75·N_ref):

VariantNoise estimateStrengthRelative cost
CA-CFAR (default)mean of reference cellsoptimal in homogeneous noise1×
GO-CFARmax of two half-windowsrobust at clutter edges~1.1×
SO-CFARmin of two half-windowsbetter in multi-target, weak at edges~1.1×
OS-CFARk-th order statisticrobust with interfering targets2–3×

The threshold multiplier α is variant-correct: CA/GO/SO use α = N(Pfa^(−1/N) − 1);

OS uses the Rohling order-statistic relation Pfa(α) = k·C(N,k)·B(α+N−k+1, k) solved

by bisection (osCfarAlpha), validated by a Monte-Carlo Pfa-recovery test. GO/SO/OS

each have a unit test demonstrating their characteristic behaviour (GO suppresses a

clutter-edge false alarm; OS rejects in-window interferers that mask CA).

2.1 Direction finding (monopulse azimuth refinement)¶

A scanned beam only knows a target is somewhere in the beam — its azimuth defaults

to the beam center. Amplitude-comparison monopulse does better: two beams squinted

±squint off boresight give a sum/difference ratio r = (R−L)/(R+L) that is monotonic

in the target's intra-beam offset, so inverting r recovers the angle to a fraction of

a beamwidth. radarv2 models this end to end:

  • the RxSynthesizer imprints the dominant in-beam target's ratio on the IqFrame

(mono_az), carried through RangeProfile → RangeDopplerMap;

  • the CfarDetector inverts it through a loaded calibration table (MonopulseTable,

param df_table) to set azimuth = beam_azimuth + table.angleFromRatio(mono_az).

The table is a loadable binary asset (ASSET-01), like the DTED tile — one of three

such tables radarv2 ships: the DTED terrain tile (map_provider), the

matched-filter reference bank (pulse_compressor, param matched_filter_table,

see 04-Signal-Processing-Pipeline), and this monopulse DF table. Each has a

build-time generator under examples/radarv2/scripts/ and degrades gracefully to the

analytic/beam-center path when absent. The full algorithm + math lives in the

radarv2_cfar_detector and radarv2_utils package READMEs.

3. Tracking (implemented)¶

The single-radar tracking pipeline is shipped (RT0–RT6): detections are conditioned

into plots and carried through association, filtering, and lifecycle management to

confirmed tracks. The chain, per node (full math + citations in each package README):

  • radarv2_plot_conditioner — DBSCAN-clusters the DetectionReport, consolidates

each cluster to one plot at its magnitude-weighted centroid (scalars from the peak

cell), with SNR and zero-Doppler clutter gating.

  • radarv2_tracking — ellipsoidal gating + global-nearest-neighbour assignment

(Hungarian/Munkres) associates plots to tracks; the live filter is an IMM over a

shared 4-state [E,vE,N,vN] bank (CV / high-q "CA" / coordinated-turn), with

alpha-beta (Kalata 1984) and Kalman-CV baselines available in filter.hpp.

  • radarv2_track_manager — owns the polar→ENU measurement conversion (Jacobian-

propagated covariance), drives the M-of-N lifecycle (Tentative → Confirmed →

Coasting → Deleted), and snapshots the IMM state into the Track / TrackReport.

Multi-radar fusion (FD-4) sits on top: radarv2_fusion /

radarv2_fusion_center fuse per-radar world tracks with Covariance Intersection

(default) or an information filter, into a CommonPicture; radarv2_control_center

closes the loop by cueing the beam to the highest-priority fused threat. See

11-Multi-Radar-Fusion.

Why this lives downstream of the signal chain

radarv2's core contribution is the signal-processing chain; tracking and fusion

are built on top of it and are now fully wired and tested. The realistic-vehicle

work (FD-1) and the IMM pair naturally: maneuvering jets are exactly what an IMM

tracker exists to follow. See FUTURE-DEVELOPMENT.
