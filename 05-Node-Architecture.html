Node Architecture¶

Scope

The aros composition: the eleven nodes of the single-radar spine, how they group

into executors and DKMs, which transport carries each link, and how the deployment

is wired. This is the blueprint the implementation follows. (A multi-radar fleet

instantiates this graph per radar and adds a FusionCenter + ControlCenter per

radar — see 11-Multi-Radar-Fusion.)

aros primer (see [[ARCHITECTURE|aros architecture]])

A node is a passive flow: it declares advertise<T> / subscribe<T> and

typed handlers, owns no thread. An executor owns one OSAL task + one inbox and

dispatches its co-located nodes in declared order. A NodeManager loads a

per-DKM deploy manifest, provisions executors, wires topics, and drives the

5-state lifecycle. Topology is fixed at boot; parameters are runtime.

`MapProvider` loads the physical maps (FD-2 → demo)

MapProvider loads real geographic binaries — DTED terrain tiles and

road/coastline vector maps — at on_configure and publishes them once as a

latched MapLayer. Three consumers use it:

RxSynthesizer (terrain-shaped ground clutter), CfarDetector (terrain

masking + line-of-sight horizon → occluded targets drop out), and HostEgress

(the physical map rendered under the tracks in Lichtblick). Proper portable

asset loading is the aros backlog item ASSET-01; until it lands MapProvider

uses an interim Linux-only loader, clearly marked non-portable. See

FUTURE-DEVELOPMENT FD-2.

Each node follows the example's house pattern (mirrors examples/radar):

  • two constructors — explicit X(std::string_view path) (test seam) and

explicit X(const NodeContext&) (uniform container ctor);

  • only on_configure overridden; all advertise/subscribe wiring happens there;
  • topics referenced by generated symbol (radarv2::topics::*), never string literals

(except the /aros/tick subscription);

  • a pure transform method (e.g. compress(), detect()) so unit tests call the

algorithm directly without booting a manager;

  • AROS_REGISTER_NODE(class, "type_key") at file scope.

2. Executors and DKMs¶

Nodes are grouped into DKMs (deployable processes). The RX DSP chain is

co-located in one executor so the per-chirp messages hand off in-cycle with no

queue hop; scan/nav/map, detection, tracking, host, and viz are separate processes to

exercise the cross-process transports. The diagram below shows the original

receive-to-host core; the shipped spine adds a trk_dkm (PlotConditioner +

TrackManager) and a viz_dkm (TargetSink) after it.

The three RX nodes share one executor so IqFrame and RangeProfile move by

DirectHandoff (staged copy-in, drained same cycle, sync: same-cycle).

RangeDopplerMap (~16 KB) crosses to det_dkm over SHM (the production

reflective-memory stand-in). DetectionReport crosses to host_dkm over TCP

(the consumer listens; the producer is the reconnecting client). ScanCommand and

PlatformNav fan out over UDP.

Stack sizing

The executors that serialize RangeDopplerMap (16 KB) and TargetReport into a

stack buffer for the SHM/TCP transports need an enlarged stack

(stack_bytes ≥ 262144), the same pattern the radar example uses for its 47 KB

TrackedObjects TCP frame.

3. Transport rationale¶

radarv2 deliberately uses all four shipped transports so the example documents the

placement-degenerates-transport rule end-to-end:

LinkTransportWhy
RxSynthesizer → PulseCompressor → DopplerProcessorDirectHandoffCo-located hot chain; per-chirp; zero-copy same-cycle
ScanCommand, PlatformNav fan-outUDPSmall control/telemetry, best-effort, multi-consumer
MapProvider → Rx/Cfar/Host (MapLayer)SHMLarge (~10 KB) static map grid; published once (latched), shared read
DopplerProcessor → CfarDetectorSHMLarge (16 KB) cross-process map; mirrors on-card reflective memory
CfarDetector → HostEgressTCPReliable delivery of the detection report to the "host"

(If two RX nodes were ever split onto different executors, the same-cycle links

would become LocalChannel — a manifest change, not a code change.)

4. Lifecycle and dispatch¶

All nodes are passive. The NodeManager in each DKM drives them through

Unconfigured → Inactive → Active. In on_configure each node reads its parameters

and sets up its publishers/subscriptions; from Active, the executor dispatches

inbound messages to the typed handlers in FIFO order.

The drive clock is the scan_dkm Tick executor: each tick advances

ScanController's scan state machine by one step and NavSource's platform model,

emitting a ScanCommand (and periodically a PlatformNav). Everything downstream is

event-driven — a node runs only when a message arrives. frame_start on the

dwell-boundary ScanCommand is the slow-time reset that tells DopplerProcessor a

new CPI has begun.

For tests, a single node is exercised through its pure transform method (no manager,

no OS thread), exactly as the radar example does — see 06-Message-Catalog for the

message structs the tests build by hand.

5. Deployment artifacts¶

Following the example conventions, radarv2 ships:

  • topics/radarv2_topics.hpp — hand-authored topic-address constants. Each is a

rootless template "/<CODE>/<node>/<leaf>"; at runtime a node calls

radarv2_utils::nsResolve(path(), topics::X), which prepends the node's radar

namespace, so the same template yields a distinct topic per radar instance (no shared

root, no codegen step).

  • packages/<pkg>/launch/<node>.launch.yaml — per-node-type interface fragments

(publishes/subscribes + param_schema).

  • launch/radarv2.launch.yaml — the single-radar deployment spine: includes the node

fragments, vars (tick_ms, inbox_depth, scenario, …), the demo variant, the

DKMs, executors, and the links with their transport + sync settings.

launch/radar.base.launch.yaml is the namespaced single-radar template the multi-radar

spine generator includes once per fleet instance.

  • config/params/<node>.param.yaml — per-node-type parameter schemas (aros.params/v1),

in the Autoware-style config/ hub alongside config/radars, config/fleets, and

config/scenarios.

aros launch build flattens the spine into per-DKM .manifest files (executors,

dispatch order, links, params) that each NodeManager loads at boot. The build needs

three --msg-dir flags: radarv2 msgs, the aros std catalog, and the param schemas.
