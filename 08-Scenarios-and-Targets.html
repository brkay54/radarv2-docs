Scenarios and Targets¶

Scope

How a radarv2 scenario defines the air picture the radar "sees", the target model

that feeds RxSynthesizer, and the seam that lets realistic typed vehicles (jets,

etc.) drop in later — the design hook for FD-1. The realism

rationale and validated physics behind the choices below live in

the simulation-realism brainstorm.

1. Scenario format¶

radarv2 reuses the example's build-time scenario codegen: each

scenarios/*.scenario.yaml (kind aeris.scenario/v1) is compiled into a static,

dependency-free radarv2_scenarios table that RxSynthesizer (targets) and

NavSource (ownship platform) consume identically. A scenario declares the platform

trajectory and a set of targets:

---
kind: aeris.scenario/v1
name: jet_intercept
title: "Fighter intercept run (weaving ingress)"
description: "A fast inbound fighter weaving toward the platform"
duration_s: 60.0
seed: 20260621          # reproducible noise + Swerling fluctuation
time_scale: 1.0         # sim-seconds per nominal time unit
platform:
  start: [0.0, 0.0, 20.0]   # ENU origin (m); mast height 20 m
  waypoints: []             # static site for this scenario
targets:
  - id: 1
    name: BANDIT-1
    type: fighter           # vehicle class → envelope + RCS + Swerling defaults
    motion: smooth          # cv | smooth (Catmull-Rom) | dynamic (g-limited, FD-1)
    spawn_s: 0.0
    despawn_s: 60.0
    waypoints:
      - { t: 0.0,  e:  2000.0, n: 18000.0, u: 3000.0 }
      - { t: 20.0, e: -1500.0, n: 12000.0, u: 2200.0 }   # weave left
      - { t: 40.0, e:  1500.0, n:  6000.0, u: 1200.0 }   # weave right
      - { t: 60.0, e:  -500.0, n:  2000.0, u:  600.0 }
---

The codegen emits POD aggregates (Waypoint, ScenarioTarget, ScenarioData) plus

an inline eval_path(waypoints, n, t, motion) returning a target's PathState

(position + velocity) at any time. Three motion models are available:

  • cv — piecewise-linear (segment-constant velocity, discontinuous at

waypoints). Correct for straight / tangential passes.

  • smooth — a Catmull-Rom / Hermite spline through the waypoints: position

passes through every waypoint and both position and velocity are continuous at

interior waypoints, so a maneuvering jet produces a smooth Doppler history instead

of the polyline's unphysical instantaneous heading snaps. This is the headline

realism upgrade over the template (see

the brainstorm).

  • dynamic (FD-1) — a g-limited flight-dynamics integrator: the cv route

velocity is treated as a command the vehicle follows within its physical envelope

(bank-limited turn rate ω ≤ g·max_g / speed so lateral accel never exceeds

g·max_g, plus longitudinal-accel and climb-rate caps). A sharp command corner the

spline would take with impossible curvature is rounded to the tightest turn the

airframe allows — the flown path lags the waypoints and traces a plausible arc.

Limits are per-type (vehicleLimitsForType, mirroring gen_scenarios.py's

TYPE_ENVELOPE), so a fighter out-turns an airliner. Pure/deterministic

(re-integrated from the path start at a fixed step). The maneuver_demo scenario

showcases a fighter commanded into a ~40 g break and held to its 9 g limit.

gen_scenarios.py also fills per-target aspect-RCS (nose / beam / tail dBsm) and

a Swerling case from a per-type physical envelope, and lints every

trajectory against that envelope (speed / load-factor / altitude) — a "fighter" doing

Mach 6 or pulling 288 g is flagged at build time (--strict turns warnings into

errors). NavSource and RxSynthesizer share the table, so target propagation and the

ownship trajectory stay in sync.

2. The target model¶

For each ScanCommand, RxSynthesizer:

  • evaluates every active target's state via eval_path at the command's timestamp;
  • computes geometry relative to the platform (PlatformNav): range, azimuth,

elevation, and radial velocity (the projection of target velocity onto the

line of sight — the quantity Doppler measures);

  • selects targets inside the commanded beam (within the HPBW of the beam azimuth/

elevation);

  • selects the target's RCS by aspect (interpolating the nose/beam/tail triple

against the line-of-sight angle, so a target flashes ~15 dB brighter broadside),

applies its Swerling fluctuation (seeded), derives a two-way SNR from the radar

equation, and injects the corresponding beat + Doppler phase into the chirp's I/Q

(see RxSynthesizer).

So a scenario target becomes a physically-placed echo whose range, velocity, aspect,

and strength all derive from its trajectory and class — which is what makes the

detection chain exercise realistic geometry (beam-aspect Doppler notch, range

walk, SNR roll-off with range).

Terrain interacts with targets (via MapProvider)

When MapProvider supplies a DTED layer, a target's

visibility also depends on the terrain between it and the platform: a

low-altitude target behind a ridge is below the radar horizon and is masked out

by CfarDetector, and ground returns become terrain-shaped clutter in

RxSynthesizer. This is what makes the sea_skimmer and (future) terrain-

following scenarios meaningful — and it pairs with FD-1's

terrain-following maneuvers and FD-2's physical maps.

3. The vehicle seam (FD-1)¶

A target's motion is sampled through the eval_path seam, which offers three fidelity

rungs: the cv polyline, the smooth (Catmull-Rom) velocity-continuous spline,

and dynamic — the FD-1 g-limited flight integrator. The type field

classifies the vehicle (fighter / airliner / helicopter / drone / missile) and seeds

its RCS aspect triple, Swerling case, and kinematic envelope; under dynamic that

envelope (vehicleLimitsForType, mirroring gen_scenarios.py's TYPE_ENVELOPE)

becomes the physical bound on the flown path: the cv route velocity is the

command, and the vehicle follows it within a bank-limited turn rate

(ω ≤ g·max_g / speed, so lateral accel never exceeds g·max_g) plus

longitudinal-accel and climb-rate caps. A command corner the spline would take with

impossible curvature is rounded to the tightest turn the airframe allows — the path

lags the waypoints, exactly as a real aircraft does. This is the

fidelity ladder's next rung above the

spline; the still-richer CV/CA/CT/Singer + closed-loop-guidance variants remain open.

Design hook preserved in v1

The implementation keeps target state sampling behind a single seam: targets are

sampled through eval_path(... , motion) at dwell time, not read as fixed

positions. The cv→smooth→dynamic progression swapped the sampler behind that

seam each time with zero change to the message catalog (and dynamic needed only

a one-line type-aware branch in RxSynthesizer) — proving the hook. The standard

target-motion models (CV / CA / CT / Singer) that the IMM tracker expects sit on the

same ladder above dynamic.
