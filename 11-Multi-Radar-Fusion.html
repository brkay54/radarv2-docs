Multi-Radar Fusion and Central Control¶

Scope

The design for running several radar instances together, fusing their

detections/tracks into one common operating picture, and **controlling all radars

centrally**. This is the FD-4

design written up as the reference. It builds directly on the

model library (each radar is a selectable profile) and on

the common world frame the base pipeline already publishes in.

As-built (FD-4a-e shipped, Linux-verified)

The fusion path described below is implemented and live, with two deliberate

departures from the original sketch, both recorded in As-built below:

  • Fusion is distributed, not centralized - every radar runs its own

FusionCenter fusing its own + every peer's world-track reports (Covariance

Intersection), publishing a per-radar CommonPicture on

/<ns>/FUSE/fusion_center/output. There is no single fusion process to lose.

  • The fleet size is dynamic (config, not compiled) - the FusionCenter has no

hardcoded radar count; it subscribes to a configured peer_tracks list and maps

each distinct transport source_id to a contributor bit. Any fleet size up to 32

(the contributor_mask width) works with zero code change.

The whole demo now runs from one config hub + a radarv2 CLI (see As-built).

0. As-built — config hub, generator, and the radarv2 CLI¶

Everything that defines a fleet lives in one Autoware-style config/ tree; a generator

turns a fleet definition into the multi-radar deploy spine, and one CLI drives the lot.

examples/radarv2/
  config/
    radars/    12 profiles - spy1 / tpy2 / smartl / gm400 / giraffe / apg81 / fps117 /
               aeris10 + the ASELSAN fleet models kalkan2 / alp300 / alp100 / eirs
               (*.profile.yaml: band, wavelength, r_max, prf, beam, scan, snr_ref)
    fleets/    library / multi5 / multi3  (*.fleet.yaml: base_tcp_port + instances[ns, profile])
    params/    per-node param schemas, incl. fusion_center.param.yaml (peer_tracks)
    scenarios/ target scripts (jet_intercept, sea_skimmer, airliner_transit, maneuver_demo)
  launch/      radar.base.launch.yaml (single-radar template, ${var} placeholders)
  scripts/
    gen_multi_radar_spine.py   fleet + profiles -> <fleet>.spine.yaml (N radar
                               subgraphs + the N*(N-1) cross-radar TCP mesh)
    radarv2                    unified CLI

The generator includes the base radar graph once per instance (namespaced by ns),

bakes each FusionCenter's peer_tracks to the space-joined list of every radar's

/<ns>/HOST/host_egress/track_output, and emits the fully-meshed TCP links from

base_tcp_port. The radarv2 CLI is the single entry point:

radarv2 list                  # what the hub offers (radars / fleets / scenarios / params)
radarv2 validate --fleet multi5   # generate spine + flatten + validate, no processes
radarv2 multi    --fleet multi5   # generate, flatten, run the N-radar fleet (+ Foxglove)
radarv2 run      --scenario jet_intercept   # single radar
radarv2 viz      --introspect DIR # Foxglove bridge against a running graph
radarv2 record   --fleet multi5   # run + record one MCAP bag per DKM into runs/<ts>/
radarv2 replay   runs/<ts>-multi5 # serve the recorded Foxglove bags to Lichtblick
Record / replay (INF3)

record runs the fleet with aros-launch-run --record (AROS_RECORD_FORMAT=foxglove),

writing one <dkm>.mcap per process into a self-contained run dir (bags/ + a copy

of deploy/); the graceful SIGTERM teardown flushes every MCAP writer before the run

is summarized. replay serves those Foxglove-native bags over HTTP and prints the

Lichtblick / Foxglove Studio remote-file URL - the fused CommonPicture

(/<ns>/FUSE/fusion_center/output) rides the viz DKM's bag with its full schema.

Verified live: multi5 -> 5 manifests + 20 cross-radar TCP links, fused

CommonPicture of ~13-14 tracks over the mesh; multi3 -> 3 manifests + 6 links

(same code, smaller fleet - proves dynamic N).

1. Why fuse¶

A single radar sees a slice of the sky: one band, one site, one coverage cone, with

blind spots (the Doppler notch, terrain masking, the horizon). Several radars of

different bands and sites cover each other's gaps. Fusion turns N independent air

pictures into one: the same target seen by an S-band volume-search set and an X-band

fire-control set becomes a single track with better accuracy, continuity through each

radar's blind zones, and an explicit record of which sensors contributed.

This is exactly the inter-radar lift of what examples/radar already does inside one

radar (plot → associate → track); FD-4 raises the same association/tracking idea to the

track-to-track level across radars.

2. The deployment shape — N instances, one network¶

Each radar is the base radarv2 node graph instantiated again with a different model

profile and site pose. Topology is static (ADR-004): the number of radars and their

wiring is a deploy-manifest choice, fixed at boot. Each radar gets a unique diagtree

namespace so its nodes and topics never collide.

Transport across radars

Within a radar, links degenerate by placement (DirectHandoff / LocalChannel) as in

05-Node-Architecture. Across radars the reports cross process/host boundaries,

so the radar→FusionCenter and ControlCenter→radar links use the cross-host carriers

(UDP/TCP, or SHM if co-hosted) — the same transport degeneration, just at the

network tier.

3. FusionCenter — track-to-track fusion¶

The FusionCenter subscribes to every radar's world-track report

(/<ns>/HOST/host_egress/track_output, its own + every peer's), lifts each into a common

frame, and resolves multi-radar duplicates into single fused tracks.

Blue = the N incoming per-radar reports · Amber = the fusion computation:

realistic-vehicle kinematics of FD-1.

Design choices — as shipped:

  • Association level — track-to-track (fuse each radar's tracks). Measurement-level

(fuse raw detections in one tracker) is more accurate but needs tight time alignment;

deferred.

  • State fusion rule — Covariance Intersection (CI), the cross-covariance-safe

rule, since each radar's tracks share unknown correlations through the common targets.

  • Time alignment — every contributing track is predicted to a common fusion

timestamp before combining (radars dwell asynchronously), keyed off the t_capture

on every message header.

  • Contributor bookkeeping — each fused track carries a contributor_mask (uint32)

and num_contributors (popcount). The FusionCenter maps each distinct transport

source_id to a contributor bit on first sight, so the mask is dynamic — no

per-radar special-casing, fleet size is configuration. The picture degrades

gracefully as the mask shrinks when a radar drops.

4. ControlCenter — the central operator seat (BUILT, FD-4)¶

The ControlCenter is the command-plane mirror of the FusionCenter: it reads the fused

air picture and commands the radars. It is built and live-verified as a closed

loop — it subscribes to the CommonPicture, picks the priority (highest-score

confirmed) threat, and emits a RadarControl that cues the radars' staring sector onto

that track's bearing, so the beam follows the fused picture instead of a fixed azimuth.

set_mode (4)switch waveform / modeScanController

Distributed, not a single box — same observable behaviour

Like the FusionCenter, the ControlCenter is **distributed: one node co-located per

radar** (control.control_center). Every radar holds the full fused picture, so each

computes the same cue and the whole fleet re-points together each cycle — the

observable behaviour of the central seat drawn above, with no single point and no

cross-host control wiring (the cue is an intra-DKM link). Collapsing all ControlCenters

into one DKM with cross-host RadarControl links — the literal single seat — is then a

manifest regrouping, not a code change (aros transport degenerates by placement).

Control respects the static-topology rule

Control changes parameters and scan behaviour at runtime — which ADR-004 explicitly

allows ("parameter change is runtime; topology change requires restart"). It does

not rewire the graph or add/remove radars at runtime; the set of radars and links is

fixed by the deploy manifest. Cueing a radar to a sector is a parameter-class change,

not a topology change.

A staring site must hold a FIXED orientation

The closed loop only confirms tracks if the staring radars do not yaw: a rotating

body frame sweeps the body-fixed staring beam's world bearing ~0.32 rad/dwell, so the

target's world position rotates and consecutive plots never associate (no track

confirms, nothing to fuse, nothing to cue). The multi-radar NavSource therefore runs

yaw_rate: 0 (the single-radar demo keeps a small cosmetic drift).

5. Messages this introduces¶

These extend the message catalog (both built in FD-4):

  • CommonPicture — the fused air picture: an array of fused tracks, each with

world-frame state (position + velocity + covariance), a track id, a quality/score, and

the contributing-radar bitmask. The single product the operator and the

3-D scene consume.

  • RadarControl — a command to one (or broadcast to all) radars: target_radar_id

(a contributor id or the broadcast sentinel 0xFFFFFFFF), a command enum

(hold/start/stop/set_sector/set_mode), the cued az_base/el_base (with an

"unchanged" elevation sentinel), mode, and the cue_track_id it points at. Carried on

a control topic from each radar's ControlCenter to its ScanController.

  • RadarHealth (deferred) — per-radar status aggregated back to the ControlCenter.

Design hook (already satisfied in v1): TargetReport is defined in a common

ENU/geodetic world frame and each radar has a unique id namespace — so fusion is a pure

transform with no per-radar special-casing. Keeping that invariant is the one thing the

base implementation must not break.
