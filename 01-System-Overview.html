System Overview¶

In one sentence

AERIS-10 is an open-source 10.5 GHz pulse-LFM phased-array radar; radarv2 is a

faithful aros software model of its receive + scan pipeline, built to exercise

the aros node/executor/transport model on a realistic radar workload.

1. What AERIS-10 is¶

AERIS-10 (upstream PLFM_RADAR) is a fully-specified FMCW X-band **phased-array

radar**. It transmits linear-frequency-modulated (LFM) chirps through a

16-element electronically-steered antenna array, mechanically rotates in azimuth

via a stepper motor, and processes echoes into range-Doppler detection maps

with target tracking. Output is GPS-absolute target positions on a map display.

The STM32F746 MCU orchestrates everything: it configures the synthesizers,

clock tree, and beamformers; sequences RF power; steps the azimuth motor; and

drives the FPGA pipeline through GPIO handshake toggles (one per chirp,

elevation change, and azimuth change). The host PC sends a RadarSettings

command to start operation and receives GPS telemetry plus the radar data stream.

2. The two variants¶

Firmware, FPGA bitstream, and timing are shared; only the RF front-end and

antenna differ.

AERIS-10N (Nexus)AERIS-10X (Extended)
Power amplifierADTR1107 integrated T/R moduleQPA2962 GaN PA + external LNA
Tx power per element1 W10 W
Antenna8×16 patch array32×16 slotted-waveguide array
Antenna gain (TBD¹)~20 dBi~30 dBi
Detection range $R_\text{max}$3 km20 km

Range scales as $R_\text{max} \propto (P_t\,G^2)^{1/4}$: +10 dB transmit power and

~+10 dB gain (each way) take 3 km → 20 km. radarv2 treats the variant as a

parameter set (transmit power, antenna gain, $R_\text{max}$), not a code

fork — the same node graph runs either variant by selecting a parameter profile.

¹ Antenna gain is upstream-TBD

The upstream parameter table marks antenna gain $G$ as TBD for both variants

(no committed number); the ~20 / ~30 dBi figures are the hardware doc's

approximate estimates, used here only to illustrate the link-budget scaling.

Same for chirp bandwidth $B$ (see 02-Physics-and-Waveform). radarv2's

parameter profile pins concrete demo values and labels them as such.

3. How radarv2 maps AERIS-10 onto aros¶

radarv2 reproduces the receive + scan behaviour as passive aros nodes

driven by executors, communicating only through typed aros messages. The table below

lists the core receive-to-host chain (seven nodes); the shipped single-radar spine adds

MapProvider, PlotConditioner, TrackManager, and a viz TargetSink (eleven nodes

total), and a multi-radar fleet adds a per-radar FusionCenter + ControlCenter. Each

node corresponds to a stage (or fused group of stages) of the real system:

AERIS-10 elementradarv2 nodeWhat it does
STM32 scan FSM + beam steeringScanControllerWalks the chirp→elevation→azimuth scan hierarchy; emits one ScanCommand per chirp with beam geometry + waveform selection; flags dwell boundaries
GPS / IMU telemetryNavSourceEmits PlatformNav (lat/lon/alt, attitude, ENU pose) — the GPSB-packet analog
Antenna + ADC + DDC (stages 1–4)RxSynthesizerSynthesizes baseband I/Q for each chirp from the scenario world: per-target range delay, Doppler, radar-equation SNR, plus noise
Matched filter + range decimation (stages 5–7)PulseCompressorCorrelates the I/Q chirp with the reference chirp; outputs a 64-bin complex range profile
Doppler processing (stage 8)DopplerProcessorAccumulates M = 32 chirps, applies a Hamming window, runs a 32-point Doppler FFT per range bin → a 64×32 range-Doppler map
Threshold/CFAR detection (stage 9)CfarDetectorCA-CFAR over the range-Doppler map → detections; converts cells to range/velocity/azimuth/elevation
FT601 egress + host transform (stage 10 + GUI)HostEgressPolar → ENU → geodetic transform; world-frame target report; terminal observability sink

The mapping is deliberately one stage ≈ one node so the documentation and the

code read against the upstream pipeline doc stage-for-stage. Details in

04-Signal-Processing-Pipeline and 05-Node-Architecture.

4. What radarv2 models vs. simplifies¶

Fidelity stance

radarv2 is an aros example, not a bit-accurate FPGA simulator. It models the

data products, message shapes, control flow, and algorithms faithfully at

tractable sizes, but does not reproduce fixed-point bit widths, clock domains,

or LVDS timing.

Modelled faithfully

  • The scan hierarchy and its timing structure (chirp/elevation/azimuth, long+short bursts, M = 32).
  • The signal-chain transforms: down-conversion → matched filter → Doppler FFT → CFAR, in the documented stage order, at the real dimensions (64 range bins, 32 Doppler bins).
  • The radar equation (two-way SNR vs. range), FMCW range/Doppler relations, and resolution/ambiguity limits.
  • Every cross-interface structure as an aros message (see 06-Message-Catalog).
  • The geometry/coordinate transform (radar polar → body → ENU → geodetic).

Simplified or upgraded

  • Fixed-point arithmetic → int16/float32 software math.
  • The placeholder fixed-threshold detector is upgraded to a proper CA-CFAR (the upstream's own documented "intended" algorithm — see 07-Detection-and-Tracking).
  • Three clock domains and CDC synchronizers are not modelled (single logical clock).
  • The host GUI/map is reduced to a world-frame TargetReport (plus tracks and, in a fleet, a fused CommonPicture), a terminal sink, and Foxglove/Lichtblick visualization.

5. Why a second radar example¶

The existing radar example (examples/radar) is an abstract

tracking demonstrator: a beam scheduler feeds a CFAR-shaped detection model into

a plot→associate→track→manage pipeline. It is excellent for the tracking and

multi-DKM deployment story but treats signal processing as a black box.

radarv2 complements it by making the signal processing itself the subject:

real down-conversion, pulse compression, Doppler FFT, and CFAR, modelled against a

concrete, fully-documented open-source radar. Together they show aros spanning the

full radar stack — raw IF to confirmed track — across two complete examples that

share conventions but exercise different layers.

6. Known upstream inconsistencies & radarv2 modeling decisions¶

A line-by-line read of the AERIS-10 source surfaced several internal

contradictions and open values. Because radarv2 aims to be a correct radar

model, each is recorded here with the decision radarv2 makes. (The upstream is

self-described alpha, with a placeholder detector and stub host code, so these

are expected.)

#Upstream issueradarv2 decision
I-1Coherent integration count. The firmware fires M = 32 = 16 long + 16 short chirps per beam position (mixed waveforms, a guard gap between them, and two different PRIs — 167 µs vs. 175 µs). Yet the Doppler processor sets DOPPLER_FFT_SIZE = CHIRPS_PER_FRAME = 32 — a 32-point coherent Doppler FFT needs 32 same-waveform, uniform-PRI pulses, which a 16+16 mixed burst cannot provide. The source never reconciles this.radarv2 models a dwell = M coherent same-waveform chirps (the physically-correct CPI for a 32-point Doppler FFT). ScanCommand.waveform_type selects the mode per dwell; the scan can run a long-mode and a short-mode pass. This keeps [[04-Signal-Processing-Pipeline\the Doppler FFT]] coherent while preserving M = 32. The upstream's mixed-burst $T_\text{pos} \approx 5.65$ ms is reported as the upstream timing, flagged as non-coherent.
I-2Chirp bandwidth $B$ is TBD. It gates $\Delta R = c/2B$, processing gain in dB, and the coupling numbers.radarv2 pins a concrete demo $B$ (≈ 150 MHz) in its parameter profile and labels every derived number a demo choice, not an upstream fact. See 02-Physics-and-Waveform §4–5.
I-3Antenna gain $G$ is TBD for both variants (parameter table commits no number).Treated as a parameter-profile value; the ~20/~30 dBi figures are flagged approximate (footnote ¹ above).
I-4USB payload mismatch. The USB-protocol doc describes a small 11-transfer per-chirp packet (header + 4 range + 4 Doppler + detection + footer); the timing-budget doc instead assumes the full 2048-cell range-Doppler map is transferred.radarv2 models the per-dwell RangeDopplerMap (2048 cells) as the egress data product (the timing-budget view), since that is what CfarDetector needs. The 11-transfer framing is documented in 06-Message-Catalog as the wire form it corresponds to.
I-5Center frequency. Firmware/README say 10.5 GHz; the GUI default is 10.0 GHz.10.5 GHz is canonical (the upstream's own resolution); the GUI value is a known stale default.
I-6GUI/tracker are stubs. The host parser, DBSCAN clustering, Kalman tracker, and map generator are pass stubs upstream.radarv2 implements the detector (CA-CFAR, 07-Detection-and-Tracking), the plot conditioner, and a shipped tracker (PlotConditioner → TrackManager, RT0–6) that confirms and maintains world-frame tracks, with a multi-radar FusionCenter fusing them into a CommonPicture.
Why this table matters

These are exactly the places a naive port would silently encode a bug. Recording

them keeps radarv2 honest: where the source is ambiguous, radarv2 picks the

physically-correct option and says so, rather than copying a contradiction.
