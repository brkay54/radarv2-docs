Reference Tables (appendix)¶

Scope

A consolidated reference of the quantitative tables radarv2 draws on — windows,

tapers, Swerling models, CFAR, pulse-compression identities, direction-finding, and

the AERIS-10 FPGA constants. Cross-doc and literature-augmented tables live here once

(so a value like the −13.3 dB uniform sidelobe or the Taylor broadening factor has a

single source of truth); the domain docs link in rather than re-state them.

Provenance legend

Each table's Source: note tags every value as one of:

[AERIS-10] authored in the upstream repo · [literature] standard radar

references (Harris 1978; Mahafza Radar Systems Analysis; Skolnik *Introduction to

Radar Systems; Richards Principles of Modern Radar*) · [radarv2] a demo modeling

choice. Disagreements between the upstream and the literature are flagged inline.

R1. Window functions¶

here is the coherent SNR penalty of windowing the matched filter — the number that

matters for detection range.*

R1b — DFT figures-of-merit (Harris 1978), for validation only. *Harris's "worst-case

process loss" (~3 dB) is a spectral-analysis quantity that bundles scalloping — it is

replace R1a's MF-loss column.*

R7. Direction finding / angle estimation¶

Upstream AERIS-10 uses beam-pointing only — radarv2 adds monopulse

The upstream documents no angle-refinement method — angle comes purely from the

commanded beam direction (≈ one beamwidth of accuracy, RCS-sensitive). The table below

is the professional-radar upgrade path (sourced from literature), and ties to the

radar model library. radarv2 has since **shipped the

amplitude-comparison monopulse row**: RxSynthesizer stamps a mono_az voltage and

CfarDetector inverts it through the loadable df_table (R9 below) to refine azimuth

to sub-beamwidth accuracy — full mechanism in 07-Detection-and-Tracking.

R9. Loadable binary assets¶

radarv2 ships three loadable binary tables, each read once at on_configure

(the same cold-path pattern across all three) and generated build-time by a script under

examples/radarv2/scripts/. The committed copies live in examples/radarv2/assets/. When

the asset is absent the node falls back to synthesizing the data in code, so the tables are

optional fidelity, not a hard dependency.

AssetLoaded byParamGeneratorWhat it is
DTED terrain tileMapProviderdted_pathgen_dted_tile.py (or parses MIL-PRF-89020B)ENU height grid → MapLayer (terrain mask, clutter)
Matched-filter reference bankPulseCompressormatched_filter_tablegen_matched_filter_table.pyconjugated chirp-reference coefficients (host analog of the FPGA chirp .mem block-RAM)
Monopulse DF tableCfarDetectordf_tablegen_df_table.pyinverse monopulse curve (ratio→angle) used to refine detection azimuth (R7)

Live wiring. radarv2 run injects all three committed assets as --param weak overrides

(the DF table is built for the single-radar 0.055-rad beam). The DF table is beam-specific,

so radarv2 multi bakes one df_table_az<half_az>.bin per distinct profile beamwidth into the

run's gen dir and wires each radar's CfarDetector to its own; the matched-filter bank is

beam-independent (one file fleet-wide). The multi base graph has no per-radar MapProvider,

so the DTED tile applies to the single-radar run only.

Source: [radarv2] examples/radarv2/scripts/{gen_matched_filter_table,gen_df_table,gen_dted_tile}.py + radarv2_utils/asset/dted.hpp + the gen_multi_radar_spine.py per-beam DF baking. Each param defaults to "" (synthesize in code) when no asset is provided.
