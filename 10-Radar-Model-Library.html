Radar Model Library¶

Scope

How radarv2 generalizes from "model the one AERIS-10 radar" to a model-driven

simulation: a library of real, recognizable professional radars — each captured as

a radar definition (a parameter profile, not code) — that the same node graph

can run as. This is the FD-3

design, written up as the reference for that work. Specs are from open literature;

every value is flagged published or assumed so the fidelity is auditable.

Status — BUILT (FD-3)

The library is realized: every radar below is a profile under

examples/radarv2/config/radars/*.profile.yaml, selectable by model name from a

fleet. spy1 / tpy2 / smartl / gm400 / giraffe / apg81 / fps117 / aeris10 (plus the

ASELSAN fleet models kalkan2 / alp300 / alp100 / eirs) all flatten through the spine

generator. The library showcase fleet (config/fleets/library.fleet.yaml) runs five

of them across four bands together; radarv2 list radars enumerates them, and the

RadarProfileLibrary ctest gates every profile's schema + instantiability.

1. The idea — a radar is data, not code¶

radarv2's nodes already read every physical constant they need from parameters and

from the ScanCommand (waveform, geometry, PRF). Nothing in RxSynthesizer,

DopplerProcessor, or CfarDetector hard-codes 10.5 GHz or a 16-element array. That

is the whole leverage point: if the band, bandwidth, beamwidth, PRF, scan rate, and

range all live in a parameter profile, then a "radar model" is just a named bundle

of those values. Swap the bundle and the same pipeline runs as a different radar.

2. The radar-definition schema¶

A model profile is a flat parameter set grouped by what it drives in the pipeline.

Each row maps a real-world spec to the equation or node it feeds (cross-references

into 02-Physics-and-Waveform).

Two derivations let beamwidth and resolution fall out of the aperture

For radars whose beamwidth is published, use it directly; for those where only the

aperture is known, radarv2 derives θ ≈ λ/D and flags the value derived. Same for

range resolution — published pulse/PRF where available, otherwise an LFM bandwidth

assumed from the radar's role (search vs. fire-control).

Realized profile fields

The shipped *.profile.yaml carries the **load-bearing physics the spine generator

threads into node params** — wavelength, r_max, prf, snr_ref_db,

beam.half_az/half_el, scan.{n_az,n_el,az_base,el_base} — plus descriptive

metadata (band, array_type, scan_type, role) and per-field provenance

comments. The richer waveform/antenna fields above (bandwidth, cpi_chirps,

elements…) are the design's full vocabulary; they fold in as the nodes grow to

read them. Adding any field is a one-file edit — a radar stays data, not code.

5. Fidelity — public vs. assumed¶

The honest part of a model library is labelling which numbers are real. radarv2 keeps

each profile's provenance per field so a reader knows what is being simulated.

DimensionPublished forAssumed / derived forradarv2 stance

Bandwidth → ΔRFPS-117 (pulses/PRF), SPY-1 (pulse range)TPY-2, APG-81 (classified); rest by rolemark all but FPS-117/SPY-1 assumed
Beamwidth → ang. resSPY-1, SMART-L, GM400, FPS-117TPY-2 (est); Giraffe, APG-81 (derive θ≈λ/D)use published; flag derived
Scan → revisitmechanical rotators (Giraffe 1 s; 5–12 rpm)SPY-1, TPY-2, APG-81 (electronic)rotators clean; electronic = dwell budget, not rpm
Range → SNR budgetSPY-1, GM400, FPS-117 (RCS-qualified)others normalized to a reference SNRuse target-qualified R_max where given
Peak powerSPY-1, TPY-2, FPS-117othersassume or normalize out (relative SNR)
The two load-bearing assumptions

Bandwidth and peak power are the hardest specs to source — most fire-control

and BMD radars classify them. radarv2 keeps both as **explicit, labelled assumptions

per model**, derived from the radar's role (search → 1–5 MHz; fire-control / C-RAM →

tens of MHz). The simulation runs either way; the label is what keeps it honest. This

mirrors the "bandwidth is the load-bearing TBD"

stance already taken for AERIS-10 itself.

6. How this feeds the multi-radar work¶

A model library is the prerequisite for running several different radars together:

once each radar is a selectable profile, a deployment can instantiate the node graph

once per radar with a different profile and site pose, and fuse the results. That is

11-Multi-Radar-Fusion. The 3-D view that shows all of them at once is

12-3D-Scene.
